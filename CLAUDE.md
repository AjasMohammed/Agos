## Project Overview

AgentOS is a minimalist, LLM-native operating system written in Rust, designed for AI agents rather than humans. Core philosophy:
- **LLMs are the CPU** — they reason and make decisions
- **Tools are the programs** — installed, versioned, sandboxed
- **Intent is the syscall** — structured declarations replace raw function calls
- **Security is non-negotiable** — capability tokens, encrypted vault, seccomp sandboxing

## Code Conventions

### Type System
- **Newtypes for IDs:** `TaskID`, `AgentID`, `ToolID`, etc. are UUID wrappers generated by `define_id!()` macro in `agentos-types/src/ids.rs`
- **Error handling:** `thiserror::Error` enums; return `Result<T, AgentOSError>` throughout — no `.unwrap()` in production paths
- **Async trait methods:** Use `#[async_trait]` from the `async-trait` crate
- **Serde annotations:** Use `#[serde(default)]` and `#[serde(skip_serializing_if = "Option::is_none")]` for optional fields

### Concurrency
- Shared state uses `Arc<RwLock<T>>`
- Polymorphic adapters use `Arc<dyn Trait + Send + Sync>`
- Shutdown signaling via `CancellationToken` (propagate it, don't drop it)
- Never block the async runtime — use `tokio::task::spawn_blocking` for blocking I/O

### Security (non-negotiable)
- Every tool execution must validate a `CapabilityToken` against the required `PermissionSet` before running
- File tools must block `..` path traversal — reject any path containing `..`
- Secrets must use `ZeroizingString` / `#[derive(Zeroize)]` — never store plaintext in regular `String`
- All security-relevant operations must be logged to `AuditLog`
- No SQL string interpolation — use rusqlite parameterized queries

### Code Style
- Prefer using existing model fields and simple approaches. Do not overengineer or introduce new abstractions unless explicitly asked. When the user interrupts, take it as a signal to simplify.

### Language-Specific Notes
- This is primarily a Rust and Python project. For Rust: use correct enum variant names (check source before guessing), handle all mutex lock sites consistently, and verify type compatibility before editing. For Python/Django: use existing model fields before creating new ones.

### Module Organization
- Each crate has a single responsibility; dependency graph flows downward (no circular deps)
- Re-export public API at crate root (`pub use`)
- Unit tests go inline with `#[cfg(test)]` modules
- Integration tests go in `tests/` subdirectories

## Testing Patterns

- After applying fixes to Rust code, always run `cargo build`, `cargo test`, `cargo clippy`, and `cargo fmt --check` before reporting completion. Do not consider a task done until all four pass.
- Use `MockLLMCore` for deterministic LLM responses — never hit real APIs in tests
- Use `tempfile` crate for filesystem isolation
- Use `serial_test` for tests that share global state (kernel boot, bus connections)
- Helper setup functions live in `tests/common.rs` (`setup_kernel()`, `register_mock_agent()`, `make_context()`)
- Test permission enforcement explicitly — confirm tools deny access without a valid `PermissionSet`
- Test security invariants: path traversal blocked, token forgery rejected, secrets cleared

## Key Architectural Patterns

### Intent Flow
```
CLI Command → BusMessage → KernelCommand → Router
  → CapabilityToken validation
  → Intent schema validation
  → Tool execution (sandboxed)
  → Result injected into ContextWindow
  → AuditLog entry written
```

### Kernel Command Handling
Commands are dispatched from `run_loop.rs` to per-domain handlers in `crates/agentos-kernel/src/commands/`. Adding a new command requires:
1. Add variant to `KernelCommand` in `agentos-bus/src/message.rs`
2. Add handler module in `crates/agentos-kernel/src/commands/`
3. Add dispatch arm in `run_loop.rs`
4. Add CLI subcommand in `crates/agentos-cli/src/commands/`

### Adding a New LLM Adapter
Implement `LLMCore` trait in `crates/agentos-llm/src/`:
```rust
#[async_trait]
impl LLMCore for MyAdapter {
    async fn infer(&self, ctx: &ContextWindow, tools: &[ToolManifest]) -> Result<InferenceResult>;
    async fn infer_stream(&self, ctx: &ContextWindow, tools: &[ToolManifest]) -> Result<InferenceStream>;
    async fn health_check(&self) -> Result<bool>;
}
```

### Adding a New Tool
Use the SDK macro or implement `AgentTool` directly:
```rust
#[tool(name = "my-tool", description = "Does X", permissions = ["read"])]
async fn my_tool(input: MyInput) -> Result<ToolOutput> { ... }
```
Or implement `AgentTool` from `agentos-tools`.

## Configuration

Default config at `config/default.toml`. Key sections:
- `[kernel]` — task limits, timeouts, scheduler settings
- `[llm]` — provider host URLs, default model
- `[vault]` — database path, key derivation params
- `[audit]` — log database path, retention
- `[tools]` — tool directories, sandbox policy
- `[scratchpad]` — agent scratchpad settings (planned: context depth, page limits)

## Platform Notes

- Seccomp sandboxing (`agentos-sandbox`) is **Linux-only** — gated behind `#[cfg(target_os = "linux")]`
- Primary development and CI target: Linux x86_64
- Wasmtime requires a compatible host for WASM tool execution

## Documentation & Planning Rule

**Every change and plan must be documented inside `obsidian-vault/` before implementation begins.** This is mandatory, not optional.

**Distribute docs across the correct folders — do NOT dump everything into `next-steps/`.** Each folder has a distinct purpose. A well-planned feature typically produces multiple files across several folders.

### Where to put it

| Type | Directory | Purpose | Example |
|------|-----------|---------|---------|
| **Feature plans (primary)** | `obsidian-vault/plans/<feature>/` | **Self-contained plan directory** — master plan, research, flow, phases, subtasks all co-located | `plans/agent-scratchpad/` |
| Simple flat design decisions | `obsidian-vault/plans/` | Single-file design rationale (1-2 phases only) | `agentos-cost-model.md` |
| Bug fixes / issues / known problems | `obsidian-vault/roadmap/` | Track what's broken and how to fix it | update `Issues and Fixes.md` |
| Reference docs for a system | `obsidian-vault/reference/` | How a completed system works (API, config, internals) | `Cost System.md` |
| Data / control flow diagrams | `obsidian-vault/flows/` | Visual flow of data or control through the system (only for standalone flows not tied to a plan directory) | `Cost Tracking Flow.md` |
| Legacy implementation checklists | `obsidian-vault/next-steps/` | Lean checklists (legacy — prefer plan directories for new work) | `08-Cost Tracking.md` |

**Default for new features:** Always create a `plans/<feature-name>/` directory. This keeps the master plan, research, flow diagrams, and all phase files co-located. Only use `next-steps/` for trivial one-off checklists that don't warrant a full plan directory.

### Complex plans get their own directory

For multi-phase features (3+ phases), create a **dedicated subdirectory** inside `plans/` that contains everything: the master plan, research, flow diagrams, and one file per phase/sub-task. This keeps related docs co-located and avoids scattering them across `plans/`, `flows/`, and `next-steps/`.

**IMPORTANT: Obsidian only shows file names, not folder paths.** Every file name must be descriptive and unique on its own — never use generic names like `main.md`, `plan.md`, `overview.md`, or `flow.md`.

#### Directory structure

```
obsidian-vault/plans/<feature-name>/
├── <Feature Name> Plan.md              # Master plan — overview, phase graph, design decisions
├── <Feature Name> Research.md          # Research/rationale (if applicable)
├── <Feature Name> Data Flow.md         # Flow diagrams
├── 01-<Phase 1 Title>.md              # Phase 1 detailed subtasks
├── 02-<Phase 2 Title>.md              # Phase 2 detailed subtasks
├── ...
└── NN-<Phase N Title>.md              # Phase N detailed subtasks
```

#### Example: Memory & Context Architecture

```
obsidian-vault/plans/memory-context-architecture/
├── Memory Context Architecture Plan.md     # Master plan
├── Memory Context Research Synthesis.md    # Source research
├── Memory Context Data Flow.md             # Flow diagrams
├── 01-episodic-auto-write.md              # Phase 1
├── 02-semantic-tool-discovery.md          # Phase 2
├── ...

```

#### Master plan file requirements

The master plan file (e.g., `<Feature Name> Plan.md`) must contain:
- **Why this matters** — the problem and motivation
- **Current state** — what exists today (table format)
- **Target architecture** — what we're building (diagram)
- **Phase overview** — table with phase #, name, effort, dependencies, and wikilink to detail doc
- **Phase dependency graph** — Mermaid diagram showing execution order
- **Key design decisions** — numbered list of architectural choices with rationale
- **Risks** — table of risks and mitigations

#### Phase file requirements

Each phase file (e.g., `01-episodic-auto-write.md`) must be **fully self-contained** — an agent can pick it up and execute it without reading any other file. It must include:
- **Why this phase** — rationale with research backing
- **Current → Target state** — what exists vs what we're building
- **Detailed subtasks** — numbered, with exact file paths, code samples, struct/type definitions
- **Files changed** — table of every file and what changes
- **Dependencies** — what phases it requires and blocks
- **Test plan** — specific test cases with assertions
- **Verification** — shell commands to confirm correctness

#### Naming rules for plan directory files

- Master plan: `<Feature Name> Plan.md` (title case)
- Research/rationale: `<Feature Name> Research.md` or `<Feature Name> Research Synthesis.md`
- Flow diagrams: `<Feature Name> Data Flow.md`
- Phase files: `NN-<kebab-case-phase-name>.md` (two-digit prefix)
- All wikilinks use the descriptive file name, e.g., `[[Memory Context Architecture Plan]]` not `[[main]]`

#### When to use a plan directory vs flat files

- **Plan directory:** 3+ phases, 5+ files, multi-week effort, needs research + flow + phases
- **Flat files:** Simple features (1-2 phases), quick fixes, single design decision

For flat files (simple 1-2 phase features), a single `plans/agentos-<topic>.md` file is sufficient. Optionally add:
- `reference/<System Name>.md` — the user-facing reference doc (written after implementation)
- `flows/<Name> Flow.md` — standalone flow diagram (only if not included in the plan file itself)

For anything with 3+ phases, always use a plan directory as described above.

### Sub-task decomposition (mandatory)

**Every plan must be decomposed into small, self-contained sub-tasks that an agent can pick up and execute independently without needing the full plan in context.**

This is critical for context window efficiency — a single monolithic plan forces the agent to load the entire document just to find the one step it needs. Instead:

1. **Create a master plan** in `plans/<feature>/` as a lean index that lists phases with wikilinks, status, and one-line descriptions. This file should fit comfortably in ~50 lines of table content.
2. **Create one file per phase** in the same `plans/<feature>/` directory, named `NN-<kebab-case-phase-name>.md` (e.g., `01-core-storage-engine.md`, `02-wikilink-parser-and-backlinks.md`).
3. **Each phase file must be fully self-contained** — it should include:
   - What to do (specific steps, not vague descriptions)
   - Which files to read and modify (exact paths)
   - What types/functions are involved (names, not just crate references)
   - Expected inputs and outputs
   - How to verify the phase is complete (`cargo test` commands, expected behavior)
   - Prerequisites: wikilinks to phases that must be done first
4. **Target 1-5 files changed per phase.** If a phase touches more than 5-7 files, split it further. The goal is that an agent can read the phase doc + the relevant source files and have everything it needs.
5. **No forward references to unwritten context.** A phase file must never say "see the master plan for details" — all details the agent needs must be in the phase file itself.
6. **Mark phase status in the master plan** as each is completed (`planned` → `in-progress` → `complete`).

Example plan directory structure:
```
obsidian-vault/plans/agent-scratchpad/
├── Agent Scratchpad Plan.md              # Master plan with phase table
├── Agent Scratchpad Research.md          # Research/rationale
├── Agent Scratchpad Data Flow.md         # Flow diagrams
├── 01-core-storage-engine.md             # Phase 1 (self-contained)
├── 02-wikilink-parser-and-backlinks.md   # Phase 2 (self-contained)
├── 03-scratchpad-tools.md                # Phase 3 (self-contained)
└── ...
```

Example master plan phase table:
```markdown
## Phase Overview
| Phase | Name | Effort | Dependencies | Detail Doc | Status |
|-------|------|--------|-------------|------------|--------|
| 1 | Core storage engine | 2d | None | [[01-core-storage-engine]] | planned |
| 2 | Wikilink parser & backlinks | 1.5d | Phase 1 | [[02-wikilink-parser-and-backlinks]] | planned |
| 3 | Scratchpad tools | 2d | Phase 1, 2 | [[03-scratchpad-tools]] | planned |
```

**Why this matters:** An agent working on Phase 3 only needs to read `03-scratchpad-tools.md` and the source files it references. It does not need to load the research, the flow diagram, or the other phases. Everything is co-located in one directory but each file is independently actionable.

### Required frontmatter

Every file must start with YAML frontmatter:

```yaml
---
title: <Short descriptive title>
tags:
  - <area>          # e.g. kernel, llm, security, cli
  - <phase>         # e.g. phase-0, v3
  - <type>          # e.g. next-steps, plan, reference, flow, bugfix
date: YYYY-MM-DD
status: planned | in-progress | partial | complete
effort: <estimate>  # e.g. 2h, 1d, 3d
priority: low | medium | high | critical
---
```
